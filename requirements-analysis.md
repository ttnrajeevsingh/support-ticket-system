# Requirement Analysis

## Selected Project Option

Support Ticket Management System (Backend-heavy)

## My Understanding

Build an internal support ticket system where users create tickets, track them through
a defined lifecycle (Open → In Progress → Resolved → Closed, with cancellation paths),
add comments, and search/filter the ticket list. The core challenge is the status state
machine — invalid transitions must be rejected server-side and the UI must only offer
valid options. Data persists across restarts via a real database.

## Functional Requirements

### Ticket Management
- FR-01: Users can create a ticket with title, description, priority, and optional assignee
- FR-02: Users can view a paginated list of all tickets
- FR-03: Users can open a ticket detail view showing all fields and comments
- FR-04: Users can update ticket fields: title, description, priority, assignee
- FR-05: Users can change ticket status only through valid state machine transitions
- FR-06: Invalid status transitions are rejected by the backend with a descriptive error

### Comments
- FR-07: Users can add a comment to any ticket
- FR-08: Comments are displayed in chronological order on the ticket detail view

### Search & Filter
- FR-09: Users can search tickets by keyword (matches title and description)
- FR-10: Users can filter tickets by status, priority, and assignee
- FR-11: Filters combine with search and work with pagination

### Status State Machine
```
Open         → In Progress | Cancelled
In Progress  → Resolved    | Cancelled
Resolved     → Closed
Closed       → (terminal)
Cancelled    → (terminal)
```

## Non-Functional Requirements

- NFR-01: API responses under 500ms for list/search with up to 1,000 tickets
- NFR-02: Data survives server restarts (PostgreSQL via Supabase)
- NFR-03: No secrets committed to repository
- NFR-04: Modular code (routes → controllers → services → repositories)
- NFR-05: Consistent error response shape: `{ error: string, code: string }`
- NFR-06: Input validated at API boundary before business logic
- NFR-07: Rate limiting on all API routes (100/min reads, 30/min writes)
- NFR-08: Type-safe end-to-end (TypeScript on both client and server)

## Assumptions

- A1: No authentication required — user selected via dropdown
- A2: "Keyword search" = case-insensitive match on title + description
- A3: Database hosted on Supabase (no local PostgreSQL install needed)
- A4: Prisma ORM for migrations and type-safe queries
- A5: Timestamps stored in UTC
- A6: UUIDs generated by the database

## Clarifications (questions for a product owner)

- Should reassigning a ticket and changing status be one action or separate?
- Is soft-delete required or is hard delete acceptable?
- Should comments be editable/deletable or append-only?
- Should search be real-time (debounced) or on submit?

## Edge Cases

- Single-character search (e.g., "t") — PostgreSQL FTS drops it as a stop word → handled with ILIKE fallback
- Status change race condition — user has stale UI, submits invalid transition → 422 returned, UI re-fetches
- Empty ticket list — meaningful empty state shown with contextual message
- Rapid typing in search — AbortController cancels in-flight requests
- Terminal states (closed/cancelled) — no transition buttons rendered, "terminal state" message shown
